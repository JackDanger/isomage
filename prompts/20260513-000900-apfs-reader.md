## Decisions

### Container-level only (volumes as directories)

Full APFS B-tree traversal (to list files within each volume) is deferred. The initial implementation mirrors the 7-Zip top-level view: the container root `/` has one directory child per APFS volume. Each child is named from the volume's `apfs_volname` field (null-terminated UTF-8 at block offset 572 within the APSB).

This keeps the implementation small and verifiable without a full omap B-tree implementation.

### fs_oid as physical block address

The NX Superblock's `fs_oid[100]` array contains OIDs for APFS volumes. For the container-level volume references, each entry is a physical block address — the APSB is read directly at `fs_oid * block_size`. This is consistent with how APFS stores container-level physical objects: their OID is their block number. Virtual/ephemeral objects would require omap resolution, but container-level volumes are always physical.

### Block size validation

Accepted range: 4096–65536, must be a power of two. Real APFS containers use 4096 (most common) or 65536. Zero, non-power-of-two, and out-of-range values return `Error::BadBlockSize`.

### APSB magic fallback

If a `fs_oid` block does not have the APSB magic at offset 32, `read_volume_name` returns `None` and `detect_and_parse` synthesises a fallback name `"volume_{oid}"` rather than failing. This is tolerant of sparse containers or containers where some volume slots are partially written.

### No hdiutil round-trip

Creating a raw (non-DMG-wrapped) APFS container without root access on macOS is not feasible in standard CI. The `apfs_detect_hdiutil` test skips gracefully on all platforms. All functional coverage is provided by hand-crafted 2-block in-memory buffers (block 0 = NX Superblock, block 1 = APSB).

### Detection signature

Seek to offset 32, read 4 bytes LE u32, check == `0x4253584e` (`b"NXSB"` in little-endian). Cursor is always restored after `detect()` regardless of outcome, matching the HFS+/NTFS/DMG pattern.
