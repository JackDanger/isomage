## Assistant

### Key decisions

**Extent tree as primary data-access path**: ext4 images built by modern
`mkfs.ext4` almost always set `EXT4_EXTENTS_FL` on all inodes. The
classical block-pointer path (direct + single/double/triple-indirect) is
preserved for ext2 images and for ext3 images that were created without
the extents compat flag. Both paths are exercised by the unit tests via the
synthetic in-memory ext2 image builder.

**`file_location` only for single contiguous runs**: The TreeNode contract
says `file_location` points to a byte offset in the image that `cat_node`
can seek to and read `file_length` bytes from. This is only safe when all
data is physically adjacent. We guarantee it for:
- Extent-tree depth=0 with exactly one leaf extent whose block count covers
  the full file.
- Classical mode where all needed blocks are direct (≤ 12 blocks) and
  happen to be physically consecutive.

Multi-extent files and files requiring indirect blocks still appear in the
tree with correct `size` and `file_location = None`. Future work can add a
scatter-gather read path for `cat_node`.

**HTree is transparent**: ext4's HTree (hashed tree-of-directories)
feature stores directory entries in the same on-disk structure as linear
directories — data blocks that contain linear `ext2_dir_entry_2` records.
The HTree index lives in additional blocks that we skip (they fail the
`inode != 0` filter or have nonsense names beginning with `\0`). Walking
all data blocks of the directory inode via the extent tree or indirect
pointers picks up the correct leaf blocks.

**In-memory ext2 image for unit tests**: Rather than shipping a binary
test fixture, the unit tests build a minimal 256 KiB ext2 image
programmatically. This keeps the repo clean, is reproducible, runs on
every platform without any external tools, and exercises the exact code
paths described in the spec. The image has:
- Superblock at offset 1024 with `s_first_data_block = 1` (1 KiB blocks).
- Block group descriptor at block 2.
- Inode table at block 5 (inode #2 = root dir, inode #3 = hello.txt).
- Root directory data block at block 6 with ".", "..", and "hello.txt"
  entries using the `INCOMPAT_FILETYPE` layout.
- File data block at block 7 with 12 bytes "hello world\n".

**Why `mke2fs -d` for the many-files and nested-dir tests**: `mke2fs -d`
(available since e2fsprogs 1.43, standard on Ubuntu 18.04+) creates an
ext image directly from a host directory tree without requiring per-file
`debugfs` invocations. This makes the "50+ files" and "500+ files" tests
simple and reliable. `debugfs -w -R "write"` is used only for the tests
that need a file injected after a base image is created (e.g. the symlink
test, which needs `debugfs -w -R "symlink"`).

**Block size tests**: We test both 1 KiB (`-b 1024`, `s_log_block_size=0`,
`s_first_data_block=1`) and 4 KiB (`-b 4096`, `s_log_block_size=2`,
`s_first_data_block=0`) to exercise both BGD table offset calculations.

**e2fsck acceptance**: The `ext4_e2fsck_accepts_image` test runs `e2fsck
-n` (read-only) and accepts exit codes 0 (clean) and 1 (corrected without
prompting). Exit ≥ 2 indicates a serious problem. This gives us confidence
that the image used in the test is actually a valid ext4 image before
isomage parses it.

**7zz cross-validation**: `7zz l -slt` produces a long-format listing with
`Path = ` and `Size = ` lines. We parse those and compare sizes for known
files. When 7zz does not understand raw ext4 (it may require the partition
be presented separately), we treat a non-zero exit as non-fatal and skip
the size comparison.

**Depth limit of 32**: To protect against corrupted images with circular
directory references, we cap directory recursion at depth 32. The kernel
supports up to 32 levels of subdirectory nesting; any image claiming
deeper nesting is suspicious.

**No fat.rs created**: The `fat = []` feature flag already existed in
`Cargo.toml`. We added `pub mod ext;` to `src/formats/mod.rs` but did not
add `pub mod fat;` because `src/formats/fat.rs` does not yet exist. The CI
changes add `ext` to the feature matrix but leave `fat` for a subsequent
PR that will implement the FAT reader.

**Symlink handling**: Symlinks (`S_IFLNK`) appear in the tree as file
nodes with their correct `size` (for short symlinks stored in `i_block`
inline, the size is the target path length). We do not resolve symlinks —
they are included verbatim, matching how ISO 9660 Rock Ridge handles them.

**What I skipped**: Did not implement inline-data file content reads
(`EXT4_INLINE_DATA_FL`). The inode's `i_block` contains the data directly,
but the implementation would require knowing the exact inline-data layout
including the `ext4_inline_data` xattr header. Inline-data files appear in
the tree with `file_location = None` and correct `size`.
