## Human

Implement an NTFS filesystem reader for the isomage Rust library, following
the same conventions as the existing `ext.rs` module. The reader should
detect by OEM ID, parse the MFT, walk attributes, and produce a `TreeNode`
tree. Detailed binary format specification provided inline.

## Assistant

### Key decisions

**MFT traversal strategy**: read records sequentially from `mft_lcn *
cluster_size` until either an unreadable/non-FILE record is hit (beyond the
system range) or the 1 million-record guard fires. This mirrors how the
ext reader walks inode tables sequentially rather than following any
in-volume index, which keeps the code straightforward and avoids any
dependency on the `$Bitmap` or `$INDEX_ROOT` structures.

**Update-sequence fixup**: applied unconditionally after each FILE record
read. A USN mismatch at a sector boundary is logged but not fatal — a
read-only parser has no obligation to enforce write-protect fidelity. The
fixup is required to get correct data from the attribute area that spans
sector boundaries.

**Namespace priority for `$FILE_NAME`**: when multiple `$FILE_NAME`
attributes exist (Win32 + DOS is the common case), we choose by namespace
priority Win32&DOS (3) > Win32 (1) > POSIX (0); DOS-only (2) is skipped
entirely because the Win32 companion is always present whenever a DOS name
exists. This produces the most human-readable names without requiring
consumers to understand the namespace field themselves.

**`file_location` for resident vs. non-resident `$DATA`**: resident data
gets an absolute image offset (MFT record start + attr offset + value
offset). Non-resident data with exactly one run gets `start_lcn *
cluster_size`. Multi-run files get `None` — this matches the ext reader's
"single contiguous run" rule and keeps `cat_node` semantics consistent
across formats.

**System record exclusion**: records 0–11 are skipped unconditionally.
Record 5 (root directory) is excluded from the flat record list and
instead used as the tree root. This avoids leaking `$MFT`, `$Volume`,
`$Bitmap`, etc. into the user-visible tree.

**Tree construction via parent-reference map**: the MFT encodes each
record's parent as a 48-bit MFT record number embedded in the
`$FILE_NAME` attribute. We collect all valid non-system records into a
`HashMap<parent_ref, Vec<RecordInfo>>` then recurse from record 5. The
depth is capped at 32 to guard against corrupted images with circular
parent references. When `TooDeep` is hit for a subtree we skip that
subtree rather than propagating the error, so a single corrupt directory
cycle does not abort the whole parse.

**Attribute walking**: we do two traversals of the attribute list in
`extract_record_info` — one via the pre-parsed `Vec<Attribute>` returned
by `parse_attributes`, and a parallel byte-offset cursor `attr_pos` that
tracks the absolute position within the record buffer. The second cursor
is needed to compute the absolute image offset for resident `$DATA`
without storing additional position metadata in the `Attribute` struct.
This is a slight redundancy but keeps each helper focused.

**`clusters_per_file_record_segment` sign convention**: positive values
are multiplied by `cluster_size`; negative values give `2^|value|` bytes
directly (the typical case: `-10` → 1024 bytes). The cast chain
`data[64] as i8` → `(-cpfrs) as u32` → `1u64 << …` is the standard
NTFS formulation and is guarded against zero and implausibly large
values.

**No `$ATTRIBUTE_LIST` follow-through**: attribute lists that span
multiple MFT records (base record + extension records) are not followed.
Attribute list parsing is complex and only necessary for very large
directories or files with hundreds of attributes — rare in practice. The
attribute list type code is explicitly matched and silently ignored so
the parser does not confuse it with unknown attribute types.

**Round-trip tests**: `mkntfs` from the `ntfs-3g` package is available on
Ubuntu CI but not on macOS brew. All tests use `require_or_skip` so they
skip silently on macOS. `ntfscp` (also from `ntfs-3g`) is required for
any test that injects user files; those tests additionally `require_or_skip`
on `ntfscp` before attempting injection. An empty-volume detection test
and an `ntfsfix -n` structural check run with only `mkntfs` present.

**Unit tests without a full image**: the in-memory image builder writes a
valid boot sector and two FILE records (root at slot 5, a user file at
slot 12) with correct update-sequence arrays. The root FILE record has
parent_ref pointing to itself (5→5), which the flat-record builder skips
as a self-referential record; the user file has parent_ref = 5 so it
appears as a child of root. Resident `$DATA` content (`b"hello ntfs\n"`)
is embedded in the user record so `file_location` and content-read tests
work without any real image.
