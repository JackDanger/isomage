## Human

Add ZIP and TAR archive readers to isomage, each behind their own feature flag
(`zip`, `tar`). The goal is to close the compatibility gap with 7-Zip for
archive formats. Also: complete the `working_dir_is_src()` support on the
`RoundTrip` builder so `zip` tool round-trip tests work correctly.

## Assistant

### Key decisions

**ZIP reader approach**: Scanned backwards from EOF for the EOCD signature
(`0x06054b50`), then read the Central Directory in one buffer. This is the
standard approach and handles arbitrary comment lengths. ZIP64 detection is via
the sentinel values (`0xFFFF`, `0xFFFFFFFF`) in the EOCD, with the ZIP64 EOCD
Locator searched 20 bytes before the EOCD position. Chose this over
forward-scanning because EOCD is the authoritative record per the APPNOTE spec.

**TAR reader approach**: Sequential block-by-block scan from position 0, which
is the only correct approach for TAR (no directory at end). Handled GNU
long-name (`L`), GNU long-link (`K`), and PAX extended headers (`x`/`g`)
inline with a `pending_name`/`pending_size` override pattern. The
`div_ceil(BLOCK)` idiom (stable since Rust 1.73, MSRV 1.74) computes block
counts without integer overflow risk.

**Leading `./` stripping in TAR**: `tar -C $SRC_DIR -cf $IMAGE .` stores all
entries with a `./` prefix (e.g. `./hello.txt`). The TAR parser strips the
leading `./` from every entry name before building the tree. Without this,
the entire tree ends up under a `.` directory node rather than at root.

**`working_dir_is_src()` implementation**: Added a `cwd: Option<&Path>`
parameter to `Tool::run_in_venue` in `tests/common/tool.rs`, threading it
through from `RoundTrip::try_build`. The ZIP tool requires running from
`$SRC_DIR` so stored paths use relative form (e.g. `hello.txt` rather than
`/tmp/.../src/hello.txt`). TAR uses `-C $SRC_DIR` instead — a TAR-native
flag — so it does not need `working_dir_is_src`.

**Tree construction**: Both ZIP and TAR use a `HashMap<String, TreeNode>`
keyed on the path-from-root (without leading `/`), then a recursive `attach`
helper wires parent/child relationships. Sorted path iteration ensures
alphabetical child order, which is what `assert_tree_invariants` checks.

**Stored-only ZIP in unit tests**: The in-memory test helper builds a real
ZIP binary structure so `detect` and `detect_and_parse` are tested without
the system `zip` tool. Method 8 (deflate) is recognized but `cat_node` for
compressed files is not implemented — only method 0 (stored) supports
`file_location`-based reads.

**What I skipped**: Deflate decompression for ZIP method 8 — this requires
a `deflate-gzippy` feature with a real decompression crate. Compressed ZIP
files are detected and their tree structure is returned, but `cat_node` will
return no bytes. Added to future backlog. Also skipped: multi-volume TARs,
GNU sparse files, and POSIX ACL extended attributes — none appear in any test
image and all are documented as not-implemented in the module docs.
