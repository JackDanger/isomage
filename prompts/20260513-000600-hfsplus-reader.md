## Assistant

### Key decisions

**Leaf-chain walk rather than full B-tree traversal**: The HFS+ catalog B-tree
stores records in a doubly-linked list of leaf nodes (each node's `f_link`
points to the next). Walking this chain visits every record without recursing
into index nodes, which avoids implementing the full B-tree search algorithm.
This is simpler to implement correctly, has no recursion depth risk, and is
sufficient for our read-only use case where we want all entries (not a specific
key lookup).

**Catalog file location via extent[0]**: The volume header's `cat_file` fork
data holds the catalog B-tree's on-disk location. We use the first extent's
`start_block * block_size` as the byte offset. If the catalog fork is empty
(logical_size == 0 / no extents), we return an empty record list rather than
an error — this covers freshly-formatted volumes with no user data where the
catalog header may point nowhere useful.

**CNID 2 as the tree root**: The HFS+ root folder always has CNID = 2
(`kHFSRootFolderCNID`). Every user-visible file and directory hangs off this
CNID (directly or transitively via subdirectory CNIDs). We seed the folder map
with `(CNID 2 → "/", parent 0)` and build the rest from catalog records.

**Topological attachment by repeated passes**: After collecting all folder
CNIDs and file records, we attach subdirectories to their parents by iterating
up to `folder_map.len() + 1` times, each pass attaching any child whose parent
already exists in the node map. This handles arbitrary depth without a
recursive algorithm, and the iteration bound prevents infinite loops on
corrupted link cycles.

**`file_location = Some(offset)` only for single-extent forks**: Files whose
data fork fits entirely in one contiguous run of allocation blocks get a
`file_location`, enabling `cat_node` to serve them. Multi-extent files get
`None` — implementing overflow extent tree resolution would require reading the
extents overflow B-tree, which is out of scope for the initial reader.

**UTF-16 BE decoding**: All HFS+ filenames are stored as UTF-16 big-endian
(HFSPlusUniStr255). We decode with `String::from_utf16_lossy` after collecting
the code units, replacing invalid surrogates with U+FFFD. This matches the
existing GPT module's lossy UTF-16 approach.

**Round-trip tests use `mkfs.hfsplus` (Linux only)**: The `hfsprogs` package
provides `mkfs.hfsplus` on Debian/Ubuntu. macOS has `newfs_hfs` but it
requires a block device path — the RoundTrip builder writes to a regular file,
so `newfs_hfs` fails. We declare the tool as `Tool::new("mkfs.hfsplus")`
without aliases, so the tests skip cleanly on macOS. We added `hfsprogs` to
the Ubuntu apt install list in CI.

**`detect` restores the cursor**: Following the project convention established
by the GPT/MBR parsers, `detect` saves the stream position, reads the 2-byte
signature at offset 1024, then restores the position regardless of success or
failure. The cursor restoration is unconditional (even on I/O error for the
seek-back, we eat the error rather than shadow the original result).

**No unsafe, no deps**: The implementation is pure safe Rust using only `std`.
All multi-byte fields are decoded with `u16/u32/u64::from_be_bytes` — no
transmute, no packed structs. The `hfsplus` feature flag is already declared
in `Cargo.toml` from the v3.0 feature matrix; we just wire up the module.

**What was skipped**: File content reading beyond `file_location` population
(multi-extent files, overflow B-tree). Extended attributes. Resource forks.
Symlinks (Rock Ridge equivalent). Journal replay. These are non-goals for the
initial reader; the leaf-chain walk surfaces the directory tree shape, which is
what downstream tools need.
