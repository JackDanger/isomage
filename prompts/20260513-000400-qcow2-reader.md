## Human

Implement a QCOW2 (QEMU Copy-On-Write v2) reader for the isomage Rust library.
The reader should follow the same pattern as `src/formats/vhd.rs` (big-endian
container, `detect` + `detect_and_parse` public API, `TreeNode` output).

Key requirements:
- `detect<R: Read+Seek>` checks magic `0x514649fb` at offset 0, restores cursor
- `detect_and_parse<R: Read+Seek>` parses the 72-byte common header (v2 and v3)
- Error variants: `TooShort`, `BadMagic`, `UnsupportedVersion(u32)`,
  `Encrypted`, `BadClusterBits(u32)`, `Io(io::Error)`
- TreeNode output: root `/` dir with one `disk.qcow2` file child,
  `file_location = None` (L1/L2 table indirection), `file_length = Some(disk_size)`
- 7+ unit tests using a hand-crafted 72-byte v2 header
- Integration tests in `tests/qcow2_round_trip.rs` using `qemu-img`
- Gate on `#[cfg(feature = "qcow2")]` (feature already exists in Cargo.toml)
- Add `qcow2` to all CI feature lists alongside `vmdk` and `vhd`

## Assistant

### Key decisions

**Header size**: Read exactly 72 bytes for the common v2/v3 header. The v3
extension fields (incompatible_features, compatible_features, etc.) start at
byte 72 and are not needed for detection or basic parsing, so we deliberately
stop there. This keeps the reader simple and avoids having to handle the case
where a v3 header might extend into unallocated space.

**file_location = None always**: Unlike VHD Fixed images (contiguous data at
offset 0), QCOW2 data is always addressed through L1/L2 cluster tables with
copy-on-write semantics. There is no single contiguous byte range to report.
Setting `file_location = None` consistently is both correct and safe.

**No L1/L2 traversal**: The task specifies we only need to report `disk_size`
from the header, not to actually read virtual disk data. L1/L2 traversal would
add significant complexity (cluster sizing, refcount handling, compressed
cluster decompression) with no benefit for the current parser scope.

**Validation order in read_header**: Magic first, then version, then
cluster_bits, then encryption_method. This matches the natural byte order in
the header and gives the most informative error for malformed images
(BadMagic before UnsupportedVersion before BadClusterBits before Encrypted).

**detect() reads only 4 bytes**: The detect function is called in contexts
where we want to quickly reject non-QCOW2 files. Reading 4 bytes for the
magic is sufficient; full header validation happens only in detect_and_parse.
This matches the pattern in vhd.rs (which reads only 8 bytes for detection).

**Integration test for encrypted images**: Rather than testing encrypted
rejection with qemu-img (which requires secret object plumbing that varies
across QEMU versions), the encrypted case is covered thoroughly in unit
tests. The integration tests focus on the common path (unencrypted v2/v3).

**CI updates**: Added `qcow2` to all five places in ci.yml where the
feature matrix is listed: `Build (selected feature combos)`, `Test (all
v3.0 features enabled)`, `Doc-tests`, `Build round-trip tests`, the Ubuntu
and macOS round-trip run loops, the Docker pinned round-trip job, and the
MSRV `cargo build --lib` loop.
