### Why pull fuzz forward from Phase 6

Filesystem parsers have a long CVE history from malformed input
(libarchive, ntfs-3g, btrfs). The original plan put `cargo-fuzz`
in Phase 6 (hardening), after every format reader had landed.
That order means each format reader sits for weeks without
adversarial-input coverage. Pulling it forward to the substrate
phase makes "add a fuzz target" mechanical for the agent
implementing each format â€” they copy
`fuzz/fuzz_targets/mbr_parse_sector.rs` (15 lines) and substitute
their parser. The corpus-management story is documented; the
seed-file pattern is established; the CI build-only gate catches
API drift.

### Workspace surprise

`fuzz/Cargo.toml` is a separate crate (cargo-fuzz convention).
Without explicit workspace boundaries, modern cargo (1.95) auto-
discovers it from the parent and tries to pass `--features
mbr,gpt,...` to *every* package in the inferred workspace,
including `isomage-fuzz`, which doesn't declare those features.
That blows up the parent crate's `cargo clippy` / `cargo test`
runs with "package 'isomage-fuzz' does not contain these
features".

Fix is two-fold: belt and suspenders.

1. `Cargo.toml` (parent) gets `[workspace] members = ["."] exclude
   = ["fuzz"]` so cargo from the repo root resolves features
   against `isomage` only.
2. `fuzz/Cargo.toml` gets `[workspace] members = ["."]` so cargo
   from inside `fuzz/` recognises it as a standalone workspace
   root and doesn't try to walk up to the parent.

Took an iteration to track down because shell CWD persisted
across tool calls and made the failure look like the workspace
declarations weren't taking effect (cargo was being run from
`fuzz/`, not the repo root).

### CI integration

`.github/workflows/ci.yml` gets a new `fuzz-compile` job:

- Runs on `ubuntu-latest`
- Installs nightly Rust + `cargo-fuzz` via `cargo install --locked`
- Runs `cargo +nightly fuzz build` (compiles all targets;
  doesn't run them)

We don't *run* the fuzzers in CI because libFuzzer needs tens of
seconds of warmup before it finds anything interesting, and
GitHub Actions minutes are finite. A separate `fuzz-nightly.yml`
workflow that runs each target for N minutes per night is on the
plan but not yet wired.

### Defense-in-depth: package exclusion

The CI `package` job's `forbidden_re` deny-list now explicitly
covers `tests`, `fuzz`, `HANDOFF.md`, and `Dockerfile*`. The
parent `Cargo.toml`'s `include = [...]` allow-list already
excluded these (it only whitelists `/src/**/*.rs`, `Cargo.toml`,
`Cargo.lock`, `CHANGELOG.md`, `LICENSE`, `README.md`), but the
deny-list re-asserts the same constraint so a future tweak to
`include` doesn't accidentally publish test code.

### What I didn't do

- **No actual fuzz runs.** cargo-fuzz needs nightly; local
  install is heavyweight. Manifest parses (`cargo metadata`
  validates the targets are recognised). CI's nightly job will
  catch any compile failures on first push.
- **No committed seed files.** Both `fuzz/seeds/<target>/`
  directories are empty placeholders. Real seeds (a valid MBR
  byte blob, a valid GPT header blob, known-adversarial inputs)
  are useful but generating them by hand is tedious; the
  follow-up will be a `make fuzz-seeds` target that uses
  `printf` / `dd` to build them deterministically.
- **No `fuzz-nightly.yml` workflow.** Will land separately when
  enough format targets exist to make a nightly campaign
  worthwhile.
