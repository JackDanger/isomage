## Human

Implement a WIM (Windows Imaging Format) reader for the isomage Rust library.

Public API: `detect<R: Read+Seek>(r: &mut R) -> Result<(), Error>` and
`detect_and_parse<R: Read+Seek>(r: &mut R) -> Result<TreeNode, Error>`.

Requirements:
- Pure-Rust, no new runtime deps, no unsafe
- Detect by magic b"MSWIM\0\0\0" at offset 0
- Parse the 208-byte header (little-endian) to find xml_data RESHDR

- Scan XML for IMAGE elements to extract names and sizes
- Build a TreeNode tree with one directory child per WIM image
- 7+ unit tests with hand-crafted WIM headers

## Assistant

### Key decisions

**XML-only approach, no metadata resource parsing**: The WIM file's content
(file metadata, directory structure, file data) is stored in compressed
metadata resources that require LZX or XPRESS codec implementations. Adding
those codecs would violate the "no runtime deps in default features" rule and
introduce significant complexity. The XML metadata blob, however, is stored
uncompressed in virtually all real WIM files (compression flag not set), and
it contains the human-readable image names and sizes we need to build a useful
tree. We return Error::Compressed when XML is compressed rather than silently
returning an empty tree.

**Minimal UTF-16 LE scanner instead of a real XML parser**: The WIM XML is a
flat, machine-generated document with no entity references, no nested IMAGE
elements, and no CDATA. A simple `str::find` scan for `<IMAGE INDEX="N">`,
`<NAME>...</NAME>`, and `<TOTALBYTES>...</TOTALBYTES>` is sufficient and avoids
pulling in quick-xml or similar. This matches the "zero deps" constraint.

**RESHDR offset_and_flags decoding**: The spec packs the file offset and flags
into a single u64 with flags in bits 63-56 and the offset in bits 55-0.
The extraction is `flags = (word >> 56) as u8` and `offset = word & 0x00FFFFFFFFFFFFFF`.
The compressed flag is 0x04 in the flags byte.

**Fallback to image_count when XML has no IMAGE elements**: If the XML blob
is present but empty or malformed (no `<IMAGE>` tags found), we fall back to
the `image_count` field from the header and generate synthetic "Image N" names.
This makes the reader more robust against unusual WIM variants.

**Tree shape**: Each WIM image is a directory node. No file_location or
file_length is set on image nodes because reading file data from a WIM
requires decompressing metadata resources, which is deferred. The TOTALBYTES
XML field is surfaced as the directory's `size` field.

**Round-trip tests via wimlib-imagex**: `wimlib-imagex capture $SRC_DIR $IMAGE image_name`
creates a well-formed WIM from a source directory. The `append` subcommand
adds a second image to test multi-image parsing. Both Ubuntu (`wimtools` apt
package) and macOS (`wimlib` brew package) install the tool; the CI already
listed `wimtools`/`wimlib` in its apt/brew install steps.
