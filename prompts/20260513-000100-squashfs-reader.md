## Assistant

### Key decisions

**Uncompressed-only scope**: The zero-dep constraint rules out any runtime codec
(gzip, lzma, xz, lz4, zstd). Rather than silently returning partial trees or
skipping compressed files, the parser returns `Error::Compressed` at the first
compressed metadata block it encounters. This surfaces clearly to callers who
can then enable a codec feature. The `UNCOMPRESSED_INODES | UNCOMPRESSED_DATA`
flags in the superblock are the gating check; if either is absent the parse
fails fast before reading any inodes.

**Metadata block navigation**: SquashFS places inodes and directory entries in
"metadata blocks" — each prefixed by a 2-byte LE header where bit 15=1 means
uncompressed and bits 14..0 hold the byte count. To navigate to a specific
inode or directory listing, we seek to the table start and skip `block_idx`
metadata blocks (reading just each 2-byte header + seeking past the data).
This avoids buffering unused blocks. The content of the target block is read
entirely into a `Vec<u8>` before parsing — SquashFS metadata blocks cap at
8192 bytes, so heap allocation is bounded.

**Root inode packing**: The `root_inode` field in the superblock packs two
numbers: `(block_idx << 16) | offset`. The block_idx says how many metadata
blocks to skip from `inode_table_start`; the offset is the byte offset within
the selected block's uncompressed content. This encoding means the root is
almost always at block_idx=0, offset=32 (after the first non-root inode that
mksquashfs places at offset 0). We decode it as `block_idx = root_inode >> 16`
and `offset = root_inode & 0xFFFF`.

**Directory entry name_size is u16, not u8**: The spec documentation provided
to me described `name_size` as a `u8` field, but the actual on-disk format
(confirmed by inspecting real mksquashfs output and cross-referencing the Linux
kernel's `squashfs_dir_entry` struct) uses `unsigned short` (u16). The name
then follows immediately with `name_size + 1` bytes (the +1 is the conventional
off-by-one — `name_size` stores `length - 1`). Using u8 would misparse the
name by one byte, producing a leading null and a truncated filename. Using u16
reads the entry header as 8 bytes (2+2+2+2) instead of 7, and name starts at
offset 8 within the entry. All unit tests and round-trip tests verify this.

**file_location logic**: Set only when the file has no fragment
(`fragment == 0xFFFFFFFF`), exactly one data block, and that block is
uncompressed (bit 24 of block_sizes[0] set). The `start_block` field in that
case is an absolute byte offset into the image, suitable for `file_location`.
Files using the fragment table (the common case for small files with mksquashfs
defaults) get `file_location = None` and `file_length = Some(file_size)`.

**In-memory test image builder**: The unit test builder constructs a minimal
but byte-correct SquashFS image: one metadata block for inodes (root dir inode
at the offset encoded in `root_inode`, file inode at offset 0) and one metadata
block for the directory listing. The file data bytes follow immediately after.
This builder computes the layout (inode table size, dir table size) at build
time so the file inode's `start_block` field can be filled with the correct
absolute offset. All 9 unit tests exercise different parser paths via this
builder.

**Recursion depth guard**: `build_tree` is recursive. A depth limit of 64
prevents stack overflow on malformed images with cyclic or extremely deep
directory structures. Exceeding the limit returns `Error::TooDeep`.

**Round-trip test strategy**: All mksquashfs invocations use `-noI -noD -noF
-noX` (no compression for inodes, data, fragments, xattrs) so our
uncompressed-only reader can parse them. The tests cover: empty filesystem,
single file, nested directories, 35+ files (multi-header directory listings),
exact byte verification via `file_location`, and Unicode filenames.
