## Human

After reflecting on the v1.0.0 release, the user asked: "Fix all of this.
Broaden the CI significantly. Release to crates as an executable as well
as a library — whatever is right." Where "all of this" refers to the
audit I produced in the previous turn: path-traversal in `extract_node`,
SIGPIPE handling for `-c`, `u64`-as-`usize` truncation in the extract

## Assistant

### Key decisions

**Single-crate publish, both `[bin]` and `[lib]` from one tag.** The
existing `Cargo.toml` already declares both; that's the right shape for
a tool this size and matches what users get today via
`cargo install isomage` and `isomage = "..."`. Splitting into
`isomage` (lib) + `isomage-cli` (bin) was rejected — it would break
existing `cargo install isomage` (cargo would prefer the now-libonly
crate and fail to find a binary) and double the publish surface for no
benefit. The crate is still small enough that one publish artifact
serves both audiences.

**Version: 1.1.0, not 1.0.1.** Adding `Send + Sync + 'static` to the
public error trait object is *technically* a backwards-incompatible
narrowing for any downstream code that wrote
`Result<T, Box<dyn Error>>` against the old signature. Combined with
two genuinely new CLI features (`--completions`, `--man-page`) and a
new public type alias (`isomage::Result` / `isomage::Error`), 1.1.0 is
the honest semver. Nothing about this release is bug-fix-only.

**Path-traversal: layered defense.** The fix has three layers:
(1) `validate_entry_name` rejects empty, `.`, `..`, and any name
containing `/`, `\`, or NUL bytes. This catches the Rock Ridge `NM` /
UDF FID injections directly. (2) `safe_join` then asserts the resolved
target `starts_with(canonical_root)`, which catches any future code
path that bypasses `validate_entry_name`. (3) The output directory is
canonicalized once at the start of `extract_node`, anchoring the check
to a resolved absolute path rather than the user-supplied string.
Picked layered over single-check because future contributors won't
necessarily remember why each layer is there — the deeper layer's
assertion will catch a regression where someone "simplifies" the outer.

Considered: zip-style `Path::components` walking with `.parent()`
tracking. Rejected: more code, more bug surface, and the explicit name
guard plus `starts_with` is already sound.

**SIGPIPE: cross-platform, in the library, not in `main`.** `cat_node`
matches on `io::ErrorKind::BrokenPipe` from the writer and returns
`Ok(())`. Chose this over the Unix `signal(SIGPIPE, SIG_DFL)` approach
because:

- No new `libc`/`signal_hook` dependency.
- Works the same on macOS, Linux, Windows (where SIGPIPE doesn't exist
  but BrokenPipe still does on closed sockets/named pipes).
- Library consumers benefit too, not just the CLI.
- The "process killed by signal" semantics of the Unix approach can
  confuse parent processes that read exit codes; clean `Ok(())` is
  more predictable.

Regression test injects a `Write` impl that returns `BrokenPipe` on
the first call and asserts `cat_node` returns `Ok`.

**`u64` length: clamp before cast, never after.** Old code did
`length as usize` up front, then used `usize` arithmetic throughout —
silent truncation on 32-bit. New code keeps `remaining: u64` and
computes each chunk as
`remaining.min(EXTRACT_CHUNK_SIZE as u64) as usize`. Because the
right-hand side is the `usize` constant `EXTRACT_CHUNK_SIZE = 8 MB`,
the `as usize` cast is provably safe on both 32-bit and 64-bit.

**Public error type: `Box<dyn Error + Send + Sync + 'static>` aliased as
`isomage::Error`.** Minimum responsible improvement. A typed `enum`
error would be cleaner but is the right move for 2.0, not 1.1. The
alias gives consumers a stable name to import (`isomage::Result<T>`)
without committing the crate to an enum shape.

Cascading change: the internal `Result` aliases in `iso9660.rs` and
`udf.rs` are now `use crate::Result;` rather than per-module
non-Send/Sync aliases. All existing internal errors (`io::Error`,
`String`, `&'static str`) already satisfy Send+Sync, so this compiles
unchanged.

**Doc comments on every `pub` item.** v1.0 set the API in stone with
near-zero rustdoc. v1.1 carries doc comments on every public function,
struct, field, and method, plus a crate-level overview, plus five
doc-tests (four `no_run` examples and one runnable `find_node` test).
Treating `cargo doc --no-deps -D warnings` as a CI gate from this PR
forward keeps it that way.

**`--completions <SHELL>` and `--man-page` as `exclusive` flags.**
Chose hidden-but-discoverable: they're visible in `--help` but
documented to describe what they do and where to pipe them. Using
clap's `exclusive = true` plus `required_unless_present_any` on
`file` means `isomage --completions bash` doesn't demand an `IMAGE`
argument, while `isomage` alone still gives a clean "missing FILE"
error from clap.

Chose flags over a subcommand. Subcommands restructure the entire CLI
grammar; users with muscle memory for `isomage IMAGE` would have to
care. Flags preserve every existing invocation byte-for-byte.

**clap_complete + clap_mangen vs. build.rs.** Did runtime generation
(no `build.rs`) so the artifacts are reproducible at install time
without a Rust toolchain. The release workflow can run
`isomage --man-page > isomage.1` after `cargo build` to ship a static
man page; users without the binary built locally can still pipe their
completions of choice into the right directory.

**`rust-version = "1.74"` in Cargo.toml.** clap 4 needs ≥1.74 for the
current derive features. Declaring it explicitly means downstream gets
"requires Rust 1.74" instead of cryptic trait-resolution errors.

**`include = [...]` in Cargo.toml.** Prevents the publish from
shipping `prompts/`, `.claude/`, `scripts/`, `test_data/`, `Makefile`,
worktrees, or generated artifacts. Crates.io upload is now ~10 KB of
source instead of ~100 KB of everything. (Without this, `cargo
publish` was about to ship every prompt log to crates.io — a silent
mistake.)

**Clippy fixes were not behaviour changes**. Three lint hits — the
`if_same_then_else` in `iso9660.rs` was actually a real readability
improvement (two identical branches were collapsed into one with
better comments citing the spec section). The two
`manual_range_contains` and one `manual_div_ceil` were idiomatic
modernizations only.

**`cargo fmt --all` ran across the whole tree.** ~900 lines of pure
whitespace/wrapping changes. Doing this in the same commit as the
content changes is noisy, but doing it later is a separate noisy
commit either way and CI can now permanently gate it.

### What I skipped

- A typed `IsoError` enum. Deferred to 2.0. The `Box<dyn Error +
  Send + Sync>` alias gives the same callsite ergonomics without
  locking the crate into an enum shape.
- Rock Ridge permissions, symlinks, and timestamps on extracted
  files. Real features, deserve their own design pass.
- A `--no-progress` flag. The 100 MB threshold for progress reporting
  is currently hardcoded; trivial follow-up.
- `--list` as an explicit flag (alias for no-flag list mode). Not
  strictly needed; the no-flag form is the established interface.
