behind the `fat` Cargo feature. Companion changes:

- `src/formats/mod.rs` — `#[cfg(feature = "fat")] pub mod fat;`
- `Cargo.toml` — `[[test]] name = "fat_round_trip" required-features = ["fat"]`

### Why FAT first

7z reads FAT images natively. FAT is the lowest-friction gap:

1. **Ubiquitous** — EFI System Partitions on every GPT disk are FAT32; USB
   drives, SD cards, many ROM image sideloads. A disk-image library that can't
   read FAT partitions can't fully replace 7z for most disk-image workflows.
2. **Well-specified** — the Microsoft FAT specification (fatgen103.doc, 2004)
   is unambiguous; no rev-engineering required.
3. **No new dependencies** — the parser is pure bitfield arithmetic and seeks;
   fits behind a zero-dep Cargo feature.
4. **Prerequisite for partition dispatch** — once `raw.rs` grows a dispatch
   layer, `fat::detect_and_parse` becomes the callback for partition type
   0x0B/0x0C (FAT32 CHS/LBA), 0x04/0x06/0x0E (FAT12/16), and 0xEF
   (EFI System).

### Key design decisions

**`R: Read + Seek` generics throughout** — consistent with the PR A2 direction;
`fat::detect_and_parse(&mut MmapImage::open(path)?)` works out of the box.

**`stream_position()` to anchor base_offset** — the caller positions the reader
at the start of the FAT filesystem (byte 0 for a raw FAT image, partition-start
for a partitioned image). The parser captures that position as `base_offset` and
computes all `file_location` values as absolute offsets in the original reader.
`cat_node` therefore works correctly regardless of partition nesting level.

**FAT type from cluster count, not FS-type string** — the spec says the string
at BPB+54 (FAT16) / BPB+82 (FAT32) is purely informational. All three type
decisions follow the cluster-count thresholds from § 3.5 of the spec.

**`file_location` for contiguous files only** — FAT files are cluster-chain
linked lists. For files whose clusters happen to be physically adjacent (the
common case for freshly-written images and read-only ROM distributions),
`is_contiguous(chain)` returns true and `file_location` is set to
`cluster_abs(chain[0])`. Fragmented files appear in the tree with `size` set
correctly but `file_location = None`; `cat_node` returns an error for them. This
is documented in the module-level docstring. A future improvement could add a
multi-segment read path to `cat_node`.

**VFAT LFN reassembly** — LFN entries arrive in the directory in reverse
sequence-number order. The `lfn_pieces` accumulator collects them, then
`reassemble_lfn` sorts by sequence number ascending and concatenates the 13
UTF-16LE chars per entry, stripping the null terminator (0x0000) and padding
(0xFFFF). If an LFN sequence is present the 8.3 name is ignored.

**Depth limit 32** — guards against corrupt FAT images with cyclic directory
structures. The FAT spec doesn't enforce a maximum depth; 32 is a practical
ceiling that covers all real-world nested structures.

**No unsafe, no deps** — default-feature rule upheld. `fat` feature carries
zero runtime crates.

### FAT12 bit-packing

FAT12 stores two 12-bit entries in three bytes. Entry n occupies bytes
`⌊n·3/2⌋` and `⌊n·3/2⌋+1`; even entries use the lower 12 bits of the
resulting u16, odd entries use the upper 12 bits. The `fat_entry` helper
computes `byte_off = cluster + cluster/2` (integer division), reads two bytes,
and masks/shifts accordingly. Test `fat12_file_location_points_at_cluster_data`
verifies the decoding round-trips through a hand-built image.

### Integration tests confirm real-tool compatibility

Both `mkfs.fat` (dosfstools) and `mcopy` (mtools) were present on the dev

### What I deliberately didn't do

- **No `fat12_write` / FAT write API.** Phase 4 item; no write-path code was
  added. The `write` feature is still a no-op gate.
- **No partition dispatch in `raw.rs`.** That's PR A4 or similar: updating
  `raw.rs` to call `fat::detect_and_parse` when the partition type byte is a
  FAT variant. Keeping them separate makes each PR reviewable independently.
- **No OEM codepage 437 decoding.** 8.3 names are decoded via
  `String::from_utf8_lossy`, which substitutes U+FFFD for non-ASCII bytes.
  In practice, disk image filenames are plain ASCII; a codepage-437 table is
  deferred until a real-world need surfaces.
- **No `exfat` in this PR.** exFAT is a different on-disk format despite the
  name overlap; it will be its own module behind `--features exfat`.
