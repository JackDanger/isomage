# PR #12 FAT reader — CI failures fixed

## Issues addressed

### FAT type misclassification for small forced-FAT32 images (high)

`read_bpb` determined FAT type solely by cluster count (< 4085 → FAT12,
< 65525 → FAT16, else FAT32). A 16 MiB image formatted with `mkfs.fat -F 32`
uses large clusters (e.g. 4 KiB = 8 sectors), giving only ~4000 data clusters
— below the 65525 FAT32 threshold. The parser classified it as FAT16, then
used the fixed root-directory region (root_entry_count = 0 → 0 bytes), finding
no files.

Fix: when `root_entry_count == 0 && fat_size_16 == 0 && root_cluster_32 >= 2`,
force `FatType::Fat32` regardless of cluster count. These three BPB fields are
mandatory for FAT32 and mutually exclusive with FAT12/16 per the Microsoft FAT
specification (§ 3, § 4).

### `empty_fat16` round-trip test too small for FAT16 (high)

`mkfs.vfat -F 16` on a 4 MiB image fails on Ubuntu with "too small or too
large filesystem." With dosfstools' default cluster size for 4 MiB (2 KB
clusters), only ~2035 data clusters are created, which is below the FAT16
minimum of 4086. Changed the test to 32 MiB (gives ~8171 clusters at 4 KB
clusters — well within FAT16's 4086–65524 range).

### `fat32_nested_directory` used `.` as mcopy source (medium)

`src_root.join(".")` creates a path like `/tmp/.../src/.`. When `mcopy -s`
receives a path ending in `/.`, it attempts to copy the `.` directory entry
itself into the FAT image, which FAT forbids. Changed the invocation to list
the top-level file and subdirectory as separate arguments:

```
mcopy -s -i image src_root/file.txt src_root/subdir ::/
```

The `-s` flag still recurses into `subdir`, copying `child.txt` correctly.
